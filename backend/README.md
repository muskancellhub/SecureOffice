# SecureOffice2 Backend

## Run

```bash
cd backend
python3.11 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000
```

## Notes
- OTP is sent via SMTP if configured (`SMTP_HOST`, `SMTP_FROM_EMAIL`).
- If SMTP is not configured, OTP falls back to console output (development mode).
- Order capture notifications use SendGrid when `SENDGRID_API_KEY` is configured.
- Order recipients are tenant-scoped and admin-managed via:
  - `GET /orders/notifications/recipients`
  - `PUT /orders/notifications/recipients`
- Refresh token is stored in httpOnly cookie.
- Access token is returned in API response.
- Managed services (Bronze/Silver/Gold) are auto-seeded into `catalog_items` on startup.
- CDW sync endpoint:
  - `POST /integrations/cdw/sync-routers`
  - Catalog query endpoint: `GET /catalog?type=DEVICE&category=router`
  - Cart endpoints: `GET /cart`, `POST /cart/lines`
- OTP login endpoints:
  - `POST /auth/login/otp/request`
  - `POST /auth/login/otp/verify`

## CDW Ingestion Modes

- `CDW_INGEST_MODE=script`:
  - Requires `CDW_AGENT_COMMAND` (command must print JSON array of routers to stdout).
  - Reads `CDW_QUERY` and `CDW_LIMIT` env values.
- `CDW_INGEST_MODE=openai`:
  - Uses `OPENAI_API_KEY` and `CDW_OPENAI_MODEL`.

## Catalog + BOM Integration Note

### Existing Catalog Flow (before this change)
- `CatalogService` + `CatalogRepository` write/read a single DB table: `catalog_items`.
- PAAPI products are fetched via `app/services/papi_client.py`, normalized in `CatalogService._normalize_papi_product`, and upserted through `upsert_papi_products`.
- Catalog API is served by `GET /catalog` and `GET /catalog/{item_id}`.
- Existing cart/quote/order flows already consume catalog item IDs from that same table.

### Excel Catalog Integration (this change)
- Added Excel loader: `app/services/network_vendor_catalog_loader.py`.
- Added sync endpoint: `POST /integrations/network/sync-vendor-catalog` (+ `GET /integrations/network/last-sync`).
- Startup now attempts Excel sync through `CatalogService.upsert_network_vendor_catalog()`.
- Rows are normalized into existing `catalog_items` (no parallel catalog store).
- Excel rows are tagged in attributes with:
  - `source_type: "excel"`
  - `source_name: "network_vendor_catalog_public_pricing_meraki_extreme_skymirr_inhand"`
- Laptops/phones are excluded from Excel ingestion logic.

### Unified Catalog Access
- `CatalogService.list_items` now supports unified filters across PAAPI + Excel:
  - `vendor`
  - `product_type`
  - `source_type`
  - `source_name`
  - existing category/search/price/availability filters remain intact
- `GET /catalog/{item_id}` now supports ID or SKU lookup fallback.
- `CatalogItemResponse` now includes normalized unified fields:
  - `category`, `product_type`, `source_type`, `source_name`, `pricing_basis`, `model`, `notes`, `raw_source`.

### BOM Generation V1
- Added deterministic BOM service: `app/services/network_bom_service.py`.
- Added endpoint: `POST /integrations/network/generate-bom`.
- Input:
  - calculator result (`calculatorResult` or `calculator_result`)
  - optional preferences (`preferredVendor`, `includeUPS`, `includeLicenses`, `includeInstallation`, `includeManagedServices`, `switchPortPreference`)
- Output:
  - `line_items[]`, `subtotal`, `tax`, `grand_total`, `summary`, `assumptions[]`.
- Selection rules are deterministic:
  - category matching
  - preferred vendor if available
  - fallback to lowest-priced compatible Excel item
  - derived lines for licenses/labor/cabling/UPS when catalog SKU is unavailable

### Tests Added
- `tests/test_network_vendor_catalog_loader.py`
- `tests/test_unified_catalog_and_bom.py`

### Out of Scope (left for later)
- mailbox/email routing workflows
- Stripe/payment processing expansion
- advanced AI/RAG product selection

## Topology + Draw.io Integration Note

### Current BOM Structure (input contract)
- BOM is generated by `NetworkBomService` (`app/services/network_bom_service.py`).
- The generator returns:
  - `line_items[]` with `line_id`, `item_id`, `sku`, `source_type`, `name`, `vendor`, `category`, `quantity`, `unit_price`, `line_total`, `selection_reason`.
  - totals: `subtotal`, `tax`, `grand_total`
  - `summary` and `assumptions[]`

### Where Topology Generation Plugs In
- New service: `app/services/network_topology_service.py`.
- New API endpoint: `POST /integrations/network/generate-topology`.
- The endpoint accepts a BOM payload and returns:
  - normalized `topology` JSON
  - `drawioXml`
  - `summary` (`nodeCount`, `edgeCount`, `assumptions`)

### Topology JSON Model (V1)
- New schema module: `app/schemas/topology.py`
- Core objects:
  - `metadata` (`generatedAt`, `designType`, `assumptions`)
  - `nodes[]` (`id`, `kind`, `label`, `vendor`, `sku`, `quantity`, `iconType`, `group`, `metadata`)
  - `edges[]` (`id`, `source`, `target`, optional `label`, optional `kind`)
  - `layoutHints` (`direction`, `groups`)

### V1 Rules + Assumptions
- Always includes `Internet / WAN` node.
- Core device priority: `firewall` -> `security_appliance` -> `router` -> `cellular_gateway`.
- Access flow: Internet -> Core -> Switch -> APs/endpoints.
- If no core device exists, the service connects Internet directly to switch/access and records an assumption.
- AP/camera/sensor/antenna lines are grouped by category with quantity for SMB readability.

### Draw.io XML Generation
- `topology_to_drawio_xml(topology)` converts normalized topology to deterministic draw.io XML.
- Layout is simple and deterministic left-to-right by logical layer group:
  - WAN -> Core -> Distribution -> Access -> Endpoints -> Services
- Node icon semantics stay SMB-friendly (`internet`, `wifi`, `router`, `firewall`, `switch`, `camera`, `sensor`, `cloud`, `cellular`, `antenna`).

### Tests Added
- `tests/test_network_topology_service.py`
  - firewall/switch/AP path
  - switch/AP only fallback assumption
  - AP/camera/sensor grouping through switch
  - draw.io XML validity/stability
  - grouped quantity behavior

### Intentionally Deferred
- rack elevation or cabling path diagrams
- HA/stacking/VLAN engineering logic
- RF heatmaps and simulation
- Stripe, analytics, full AI/RAG

## Demo Lead + Design History Note

### Where Lightweight Onboarding Fits
- New demo lead capture is attached at design save/submit time via `POST /designs`.
- For authenticated users, lead data is also synced into existing tenant onboarding profile fields (`organization_name`, `admin_name`, `admin_email`, `admin_phone`) to reuse current customer context.
- For anonymous/demo calls, lead + session fields can still be persisted through the same save endpoint.

### How Dashboard / History Works
- New persistence models:
  - `design_leads` (`app/models/network_design.py`)
  - `network_designs` (`app/models/network_design.py`)
- New API routes (`app/routes/designs.py`):
  - `POST /designs` (save draft/reviewed/submitted)
  - `GET /designs` (history for current actor)
  - `GET /designs/{design_id}` (reopen design detail)
  - `POST /designs/{design_id}/submit` (submit existing design)
  - `GET /designs/ops/submissions` (ops/admin queue)
  - `PATCH /designs/{design_id}/status` (manual status progression)

### Demo Status Tracking
- Extended status set: `draft`, `reviewed`, `submitted`, `in_review`, `bom_finalized`, `proposal_ready`, `approved`, `order_decomposed`, `fulfillment_in_progress`, `installation_scheduled`, `installed`, `completed`.
- Manual transition rules are deterministic in `NetworkDesignService.ALLOWED_ADMIN_TRANSITIONS`.
- Each request now persists:
  - `status_updated_at`
  - `status_history` (`status`, `changedAt`, `changedBy`, `note`)
  - milestone dates (estimated and confirmed)
  - update notes with `visibility` (`internal` or `customer`)
  - installation assistance placeholder (`installMode`, `preferredInstallDate`, `installNotes`)
  - optional decomposition buckets from BOM (`networkHardware`, `connectivity`, `managedServices`, `installation`, `accessories`)
- Submission persists `submitted_at` and triggers a lightweight mailbox-style handoff notification.

### Demo Mailbox / Handoff
- Added `EmailService.send_design_submission_handoff(payload)`.
- Uses `DESIGN_HANDOFF_EMAIL` when configured, otherwise falls back to mock console output.

### Tests Added
- `tests/test_network_design_service.py`
  - lead creation from submission
  - design-to-lead linking across multiple designs
  - status history + lifecycle transition validation
  - milestone persistence
  - customer/internal note visibility filtering
  - ops listing for submitted requests
  - decomposition bucket mapping from BOM
  - install mode persistence
  - required lead field validation

### Intentionally Deferred
- Stripe / payment collection
- automated fulfillment/CRM orchestration
- full onboarding and business verification beyond demo lead capture
- production workflow engine / vendor ordering automation

## Quote-Centric Demo Persistence Note

### Existing DB Tables Reused
- Generated network design submissions are now synced into existing commercial entities instead of introducing another lifecycle store:
  - `quotes` + `quote_lines` for BOM-backed quote persistence
  - `tenants` + `tenant_onboarding` for company/contact lead capture
  - `orders` + `workflow_instances` + `workflow_steps` for visible demo status progression
  - `assets` for persisted design artifact linkage (`asset_type = design_artifact`)

### Mapping Added
- `NetworkDesignService` now performs table sync on save/submit/status updates:
  - BOM lines are mapped into `quote_lines` with catalog linkage when possible.
  - Design status is mapped to `quotes.status` and to an order-level demo workflow template.
  - Design metadata now stores linkage IDs: `quoteId`, `orderId`, `workflowInstanceId`, `assetId`.

### Demo Lifecycle Behavior
- Demo stages continue to be business-friendly (`draft` through `completed`), and are mirrored into workflow steps for internal ops visibility.
- On submission, lead/contact fields are synced into tenant onboarding and tenant name.

### What Remains for Later
- Stripe/payment capture and payment-gated orchestration
- production procurement automation and vendor APIs
- contract-grade workflow automation/SLA enforcement
